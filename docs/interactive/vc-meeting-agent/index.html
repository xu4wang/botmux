03会前

接入引导

按下面 7 步走完就能开第一场智能体会议（第 6 步语音发言可选）。填入你的 App ID 后，权限申请和事件订阅按钮会变成直达你应用的链接；每步完成后打勾，进度保存在浏览器本地。

0/6
1

确认平台开通状态 外部门槛

会议 Bot 事件（vc.bot.meeting_*）可能需要平台侧开通。请按所在环境的开放平台流程确认应用租户已具备该能力。

查看平台开通说明 如无开通说明链接，请联系应用管理员或平台支持。
2

填入你的 App ID

在飞书开放平台找到（或创建）你的应用，把 App ID 填到下面。后续步骤的跳转按钮会自动变成直达你应用的链接，只保存在浏览器本地。

未填写
3

申请 VC 权限 必需

三项必需 scope 一键申请；实时语音是可选能力，单独申请。申请后记得发布应用版本权限才生效。

  • vc:meeting.bot.join:write（Bot 入会）单独申请 ↗
  • vc:meeting.meetingevent:read（读会中事件/字幕）单独申请 ↗
  • vc:meeting.message:write（发会中弹幕）单独申请 ↗
  • vc:meeting.bot.realtime:write（实时语音，可选）单独申请 ↗
一键申请 3 项必需权限 先在第 2 步填入 App ID
4

订阅并发布事件 必需

在事件订阅页添加下面 3 个 bot 事件并发布版本。注意：飞书没有公开 API 能自动校验事件是否已订阅，缺事件时的症状是 daemon 静默收不到 push。

vc.bot.meeting_invited_v1
vc.bot.meeting_activity_v1
vc.bot.meeting_ended_v1
打开事件订阅页 先在第 2 步填入 App ID
5

配置 botmux 命令优先

不要把手改 bots.json 当主路径。先用 botmux setup 注册或更新 bot（写盘前会校验凭证），确认该 bot 已有 larkCliProfile，再到 Dashboard → Settings 开启「允许本机使用会议监听能力」并选择监听 bot；保存时 Dashboard 会先校验 profile 与 VC 权限，校验通过后自动启用会议监听。

命令方式

botmux setup list --json
botmux setup add --app-id cli_xxx --app-secret <SECRET> \
  --allowed-users <你的邮箱或 open_id> --cli <claude-code|codex> --json
botmux restart
高级参考：vcMeetingAgent 字段（排查用，不作为用户主路径）

JSON 参考

{
  "larkAppId": "cli_xxx",
  "larkAppSecret": "...",
  "displayName": "会议监听 Bot",
  "cliId": "claude-code",

6

开通语音发言 可选

让 agent 能在会中开口说话（实时语音 M0）需要四件事：realtime 权限、TTS 语音凭证、bot 配置、会议侧开关。只用文字/弹幕可跳过这步。

  • ① 申请可选 scope vc:meeting.bot.realtime:write（第 3 步里有单独申请链接）并发布版本；
  • ② 准备可用的语音合成服务凭证（按你的部署环境选择供应商），记录 appkey / AccessKey / SecretKey 等必要字段；
  • ③ 本机配置：botmux voice（交互式选择 TTS provider、粘贴凭证，缺 opus 编码器会自动装）→ 让本机 agent 或运维开启该 bot 的 realtimeVoice.enabled → botmux restart；
  • ④ 每场会议在会议侧打开「允许 AI 智能体发言」。
配好后可用 botmux voice status 自查。
7

开一场真会验证

发起一场测试会议，把 Bot 邀进来，逐项确认：

  • 会议侧允许智能体加入（实时语音场景额外打开「允许 AI 智能体发言」）；
  • 有人说话/发消息时监听群能收到 transcript / chat 同步（真会读门，不是只有 metadata）；
  • 选择 agent 后 @agent 能接住会议上下文；
  • 输出审批链路：text 弹幕、voice 请求的成功/失败都正确显示结果。
卡在任何一步时，对照下方「完整依赖参考」按症状排查。

04会中

            

              
运行流程

              
当前主路径是手动邀请 Bot。agent 消费、会中弹幕和语音都挂在同一场会议 session 上，结束后统一收尾。

            

          


          

            

              
              
              
              
              
            

            

              

                

                  
1
用户邀请 bot 入会
meeting_invited 到达 daemon，视为显式授权，不再二次私聊确认。

                  
2
bot 用指定 profile join
join 必须绑定当前 listener bot 的 larkCliProfile，禁止落到 lark-cli 默认 profile。

                  
3
创建监听群
监听群用于同步稳定字幕、会中聊天、参会变化、输出审批卡和 agent 讨论。

                  
4
activity ingest
push 和 polling fallback 写入同一份 per-meeting state，按 sentence_id/message_id 去重。

                  
5
ended 收尾
最终 flush、发送结束记录、清 runtime record，并用 tombstone 拦住迟到 activity 复活 session。

                

              

              

                

                  
卡片选择
监听群发 agent 选择卡。下拉只暂存，点击“确认”才生效；可选择只监听，也可选 30/60/120 秒同步间隔。

                  
跨 daemon 注入
若目标 agent 不在会议 bot daemon 内，会议 daemon 调目标 daemon 的 /api/trigger，保证注入和用户 @agent 落入同一个 chat-scope session。

                  
单 tick 同拍
同一个 timer 周期先 flush 监听群，再评估 agent 注入，避免群消息和 agent 上下文存在固定相位差。

                  
catch-up
用户在监听群 @agent 时，会先 best-effort 拉一次最新稳定会议内容，再放行用户 turn。

                

              

              

                

                  
会中弹幕不是监听群消息

                  
agent 通过 botmux vc-agent request-output --channel text 提交请求，daemon 发审批卡。授权人批准后走 lark-cli vc +meeting-message-send --msg-type text，消息出现在会议聊天/弹幕里。

                  
    
                    
  • 默认需要审批；本场可切到自动文本。
    • 
                    
  • 失败时卡片展示真实错误；ok:false 不会被误判成功。
    • 
                    
  • 发送 uuid 使用 requestId，避免重复发送。
    • 
                  

                

              

              

                

                  
实时语音由 daemon 统一执行

                  
agent 只能提交语音请求，不能直接连实时音频 WS。授权人批准后，daemon 复用会议 session 持有的 voice session，合成 PCM 后按 100ms 帧推给会议。

                  
    
                    
  • 需要 vc:meeting.bot.realtime:write 和会议侧“允许 AI 智能体发言”。
    • 
                    
  • 实时协议使用 session.create 握手，等 session.created 后才推音频。
    • 
                    
  • 失败可降级成会中弹幕或监听群文本，取决于审批选择。
    • 
                  

                

              

              

                

                  
runtime store
已开始监听的会议会保存 meetingId、listenerChatId、consumer 选择、输出策略和同步间隔。

                  
恢复策略
daemon 重启后恢复监听路由和 timers；不恢复待审批 nonce。全局开关关闭时跳过恢复。

                  
重启边界
已 ACK 未 flush 的最近窗口内容可能丢失；后续可通过 restore-time polling reconcile 补偿。

                  
僵尸保护
ended 事件和 close 都会移除 runtime record；迟到 activity 用 ended tombstone 拦截。

                

              

            

          

        


        

          

            
05工作台

            

              
Botmux 功能介绍

              
这不是单纯转发 VC 事件，而是一套会议工作台：监听、同步、选择 agent、上下文注入、输出审批和可运维配置。

            

          


          

            
已实现
会议监听群
每场会议创建或复用一个群，把稳定字幕、聊天、参会/共享状态和结束状态发进去。

            
已实现
Agent Consumer
在监听群卡片选择处理 agent，会议内容按增量注入同一 chat-scope session。

            
已实现
跨 daemon 路由
会议 bot 和处理 agent 可以在不同 PM2 daemon，注入会转发到目标 agent 自己的 daemon。

            
已实现
可配置节奏
默认 30 秒单 tick；卡片可选择 30/60/120 秒。catch-up 和 fast signal 可提前注入。

            
已实现
会中弹幕
审批通过后，bot 以会议身份向会议聊天发送文本消息，不是发到监听群。

            
M0
实时语音
支持 TTS PCM 到会议实时音频链路。默认由审批卡控制，避免 agent 随意开口。

            
已实现
全局开关
Dashboard Settings 可一键关闭本机新会议监听；进行中的会议自然收尾，重启后不会恢复。

            
已实现
监听 bot 身份
全局指定哪个 bot app 负责监听。保存时校验 VC 权限、larkCliProfile 和 bot 级开关。

            
后续
最终总结
会议结束后的全量总结应由 rolling summary 或全量 transcript 输入驱动，不用空 run 伪装完成。

          

        


        

          

            
06输出门

            

              
安全与审批

              
会议内容属于不可信输入。防护重点不是让 agent “更听话”，而是把会中对外动作收敛到 daemon 审批门。

            

          


          

            

              
默认输出策略

              
    
                
  • 文本：默认审批，可在本场切到自动文本。
    • 
                
  • 语音：默认禁止或审批；不建议在 dogfood 期开放免审。
    • 
                
  • 权限只是意愿，实际能力还受 scope、会议侧开关和 endpoint 可用性硬门约束。
    • 
              

            

            

              
只认授权用户的指令

              
    
                
  • 会议转写按 speaker 标注“授权用户/仅上下文”。
    • 
                
  • 非授权用户说出的命令只作为上下文，不作为 agent 指令执行。
    • 
                
  • 声纹归因不是密码学保证，所以审批卡仍是外发动作的硬门。
    • 
              

            

            

              
审批卡

              
    
                
  • 同意语音、改发弹幕、拒绝。
    • 
                
  • 单 channel 同时只保留一个 pending，请求超时自动拒绝。
    • 
                
  • 审批结果会回注给 agent，避免重复请求。
    • 
              

            

            

              
隐私边界

              
    
                
  • raw transcript/chat 默认不长期落盘。
    • 
                
  • 监听群是会议内容可见面，群成员必须明确。
    • 
                
  • 主动 ambient 监听暂缓，避免每用户 OAuth 和隐私撤销复杂度。
    • 
              

            

          


          

            边界
            
daemon 门能挡住“会场话术注入 agent 后经正常工具面发声/发弹幕”的主威胁；它不是进程级沙箱。若 agent 进程被视为完全不可信，需要另做沙箱和密钥隔离。

08答疑

            

              
常见问题

            

          

          

            为什么不默认监听用户加入的所有会议？
            
这需要用户态事件/OAuth、token 刷新、撤销、隐私告知和用户到 bot 的路由。相比“手动邀请 bot”只多了 ambient 便利性，MVP 阶段性价比不高。

          

          

            监听群里的 agent 为什么能回答“刚才谁提到 X”？
            
会议内容注入和用户 @agent 都落到目标 agent daemon 的同一个 chat-scope session。用户 @ 时还会先做一次 best-effort catch-up，减少“我听到了但 agent 还没 flush 到”的时序差。

          

          

            会中弹幕和监听群文本有什么区别？
            
会中弹幕通过 VC meeting message API 发到会议聊天里，所有会议参会人可见；监听群文本只在 botmux 创建的监听群里可见。

          

          

            为什么输出需要审批？
            
会议转写可能包含提示词注入，speaker 归因也不是强认证。默认审批把“对真实会议说话/发消息”的执行权放在 daemon 和授权用户手里。
